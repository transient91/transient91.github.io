A Survey of Automatic Software Vulnerability Detection, Program Repair, and Defect Prediction Techniques

Posted on 2021-07-14 Edited on 2021-08-23

企业实训论文

Automatic Software Vulnerability Detection(软件漏洞自动检测)

  根据静态分析技术，漏洞处理方法进一步分为基于代码相似度的漏洞检测方法和基于代码模式的漏洞检测方法。


Code Similarity-Based Vulnerability Detection(基于代码相似性的漏洞检测)

  又称 clone detection(克隆检测)，其核心思想是相似的程序代码具有相似的软件漏洞。四种常见的克隆类型:

  • Type-1 (T1)：克隆是相同的代码片段，可能包含空白、布局或注释的变化
  • Type-2 (T2)：克隆是允许标识符、文字、类型、空白、布局和注释变化的代码片段
  • Type-3 (T3)：克隆允许额外的修改，如更改、添加或删除语句
  • Type-4 (T4)：克隆还包括语义相同但语法不同的代码片段

Grammar-Based Clone Detection Methods(基于语法的克隆检测方法)

  源代码的表示方式决定了信息提取的上限，进一步限制了模型设计和相似性度量算法的选择，最终影响检测效果。传统的基于文本的表示方法，处理数据只涉及程序代码中注释和空格的去除，主要是基于文本相似性度量方法来检测文本级克隆。然而，基于树的表示方法通过解释器分析程序代码来处理数据，其度量算法考虑了更多的程序结构信息，从语法层面检测代码克隆。
  目前，基于深度学习的方法通过构建抽象语法树截取关键节点信息，将其转化为特征向量或哈希值，然后选择合适的神经网络模型进行训练，最后利用该模型实现克隆比较。

CCLearner

  • 提出了一种基于句子的深度学习代码克隆检测方法，定义了 8 种标记类型。
  • 首先，使用 ANTLR 解析每个方法并提取 C1–C3，并使用 Eclipse ASTParser 为每个方法创建一个抽象语法树，以进一步提取 C4–C8 令牌。然后，计算每种方法中每种类型标记的频率和方法对之间的相似度，将其输入到分类器进行训练，以检测给定代码库的克隆。
  • 由于标记只揭示了语法层面的代码关系，这种方法度量代码函数之间相似性的能力是有限的。
  • 只使用关于源代码的词法信息，不考虑代码结构信息，这使得检测功能克隆对变得困难。

CLDH

  • 提出了一种称为 CDLH 的端到端深度学习框架
  • 以有监督的方式捕获代码片段功能行为的词法和句法信息，并进行哈希，以进一步将实值表示转换为二进制哈希代码。
  • 与 CCLearner 相比，学习到的特征更加紧凑，显著提高了检测效率，节省了存储空间。

White et al.

  • 综合考虑了代码结构和标识符，同时用于建模。
  • 与 CCLearner 基于令牌频率统计的工作不同，本文使用贪婪的方式在多个树结构之间进行转换来表征代码，并使用 RtNN 模型将代码片段中相似的术语映射到相似的连续值向量。此外，该模型基于 RvNN 模型来学习不同粒度级别的代码片段。在实验集中，使用 ANTLR 对源代码进行标记化，使用 RNNLM 工具包 为每个系统训练几个 RtNNs，改变隐藏层的大小和深度。然后，用困惑度作为评价模型质量的代理，模型质量越高，精度越好。

Marastoni et al.

  • 提出了一种新的程序相似性检测方法，将二进制代码转换成图像数据，输入训练模型的 CNN 网络。
  • 同时，作者使用 Tigress C obfuscator (C 混淆器)迭代转换语法对原始数据集进行扩展，对模型进行全面训练。由于 CNN 网络需要查看从二进制文件中提取的整个图像来完成每个文件的分类，二进制可执行文件的大小限制了系统的性能。
  • 在实验阶段，CNN 中的所有权重都是用从标准差设置为 0.1 的正态分布中获取的随机值初始化的，以避免在早期阶段陷入局部极小值，并进一步微调以下参数以提高模型的性能 :test_ratio、 gradient_rate、 limit、random_seed 和 norm。

各种代码表征方法组合

  • 实现了四种代码表示 :标识符、抽象语法树、控制流图和字节码，并为每种表示训练了一个模型。实验结果表明，通过整合不同的模型，可以进一步提高代码克隆检测的准确性。

  与文本和标记方法相比，基于语法的代码表示方法考虑了程序代码的结构信息，对于程序语句的序列转换和微小的修改具有更好的容错性。然而，基于树的方法仍然面临一些缺点。随着程序逻辑结构的增强，构建程序抽象语法树的复杂度增加，进一步导致提取关键代码和向量变换操作的计算开销增加。

Semantic-Based Clone Detection Methods(基于语义的克隆检测方法)

  语义层的代码表示不仅考虑了源代码的语法，还利用了关于程序代码的控制流和数据流信息，解决了代码功能层的代码克隆检测问题。由程序生成的数据流图 DFG、控制流图 CFG 和程序依赖图 PDG 是目前主流的表示方法。
  程序有定义良好的语法，抽象语法树可以很好的表现出来。 AST 在语法级克隆检测中表现出了很好的效果，但是它不能有效地检测不连续的代码克隆。

  与一般的自然语言文本特征相比，基于语义的方法充分考虑了程序结构、序列和特殊的语法信息，深化了语义特征。然而，基于语义的表示技术需要使用生成器为每种编程语言生成程序依赖图，可扩展性差，很少谈到神经网络模型的选择和优化。
  在现有的代码相似性检测方法中，通常使用四层网络结构模型。 RNN 和 CNN 在数据关系处理和特征提取方面各有优势，其优势可以考虑设计新的网络结构。对于特征参数的选择，可以借鉴 k-fold 交叉验证的训练方法。

Code Pattern-Based Vulnerability Detection(基于代码模式的漏洞检测)

  • 基于代码模式的漏洞检测技术主要涉及两个阶段。
    1. 在训练阶段，利用控制流和数据流技术提取程序中的关键代码，借助当前主流工具 (如 word2vec)将其转化为向量，输入适当的神经网络进行监督训练。
    2. 在检测阶段，对新的软件程序执行相同的数据处理，用于通过学习的模型检测现有的漏洞。
  • 根据程序是否需要运行，基于代码模式的漏洞检测方法分为静态检测方法和动态检测方法
  • 目前用于模型训练的网络结构包括 CNN、 RNN 和 LSTM。

Static Detection Methods(静态检测方法)

  静态检测方法。 静态分析是指在不运行软件的情况下，通过构造抽象语法树和程序依赖图进行程序分析的过程，其分析对象通常是源代码或可执行代码。与可执行代码相比，源代码分析可以获得更多的语义信息，综合考虑执行路径上的信息，从而发现更多的漏洞，提高命中率。

  基于深度学习的静态漏洞检测方法在源代码层面分析程序代码的依赖性，并基于程序切片的思想对易受攻击的代码进行预处理。现有的研究中，每个样本代码的标注都需要手动进行，模型只针对特定语言和类型的漏洞，不适合扩展到大型项目。同时，不同特征参数的选取对神经网络模型的影响在文章现有的方法中没有过多的解释。

Dynamic Detection Methods(动态检测方法)

  动态检测方法。 动态分析是通过运行特定程序并获取程序的输出或内部状态等信息来验证或发现软件漏洞的过程，分析的对象是可执行代码。与静态方法相比，动态方法通过分析漏洞来获取特定的操作信息，因此所分析的漏洞通常更准确，误报率更低。

  静态分析工具已被证明在特定应用领域特别有效，例如嵌入式系统或航空航天应用。然而，在更常见的软件上使用这些技术要困难得多。

Automatic Software Program Repair(自动软件程序修复)

  早期的自动修补技术主要用于防止蠕虫的传播，随着技术的发展，自动修补技术慢慢渗透到计算机软件安全的各个方面。程序自动修复技术可以辅助软件程序中某些缺陷的自动修复，从而有效减少软件开发人员的程序调试时间。

Patching Process(修补过程)

自动修补技术分为以下三个阶段: 软件故障定位阶段、补丁生成阶段和补丁评估阶段。

the software fault location phase(软件故障定位阶段)

  软件故障定位是程序自动修复的先决条件，主要用于识别程序中潜在缺陷或漏洞的位置。目前，常用的故障定位技术分为两类: 静态故障定位技术和动态故障定位技术。

  • 静态故障定位技术主要通过程序分析技术获得被测程序代码之间的控制依赖和数据依赖关系，从而确定和定位故障位置。
  • 动态故障定位技术通过执行预选的测试用例来获得程序的执行信息，并通过分析程序的执行流程来定位缺陷语句在测试程序中的位置。

the patch generation phase(补丁生成阶段)

  构建补丁阶段通过检查程序代码结构和分析示例代码提交和修改信息来定义一些操作规则，然后通过定义的操作修改缺陷语句。

基于搜索和语义，通常使用两种补丁生成方法。

  • 基于搜索的补丁生成方法是在搜索空间中找到补丁并对其进行验证的过程，通常通过缩放搜索空间大小和优化搜索策略来提高补丁生成效率。
  • 基于语义的补丁生成方法通过集成诸如故障定位、约束求解和程序合成等技术来生成补丁。

the patch evaluation phase(补丁评估阶段)

  补丁评估阶段评估生成的候选补丁，直到找到能够让程序执行所有测试用例的补丁。

Grammar-Based Patching Technology(基于语法的修补技术)

  基于语法错误的程序修复技术通过学习代码语法特征来修改错误代码，以达到特定的语言规范。

Token-Level Learning and Repair Technology(令牌级学习和修复技术)

  计算机程序的源代码是一种类似于自然语言的语法结构的纯文本表示，其中自然语言技术和深度学习技术的发展使得基于学习的修复成为可能。
  token-level task, 即 BERT 模型预训练的两个任务之一, 等价于完形填空任务, 即给出句子中其他的上下文 token, 推测出当前位置应当是什么 token.

Text-Based Learning and Repair Technology(基于文本的学习和修复技术)

  基于文本的学习和修复技术。由于程序各部分之间的依赖性，即使是一个单一的错误也可能需要对整个程序进行分析。修复精度是程序修复技术中的一个重要指标，以前的文献基于令牌级精度定位和修复程序错误，其生成的错误补丁不能完全修复编程错误。
